# Changelog from original Carte-app PWA

Source: `github.com/haorantang97/Carte-app` (Vite + React + shadcn PWA, generated by Lovable.dev).
Target: this repo (`/Users/tanghaoran/Carte`, Expo SDK 54 RN app).

The user's brief was *"完美复现原版的功能意图,但用最合理的方式;发现逻辑漏洞或问题可以按理解修改;最后给出修改清单"*. This document is that
clinic. Every change is annotated with **what was wrong**, **what we did
instead**, and **why**. Items grouped as in the original implementation plan.

## Data layer — schema fixes

| # | Item | Before (original PWA) | After (this app) | Why |
|---|---|---|---|---|
| 1 | `dish_likes` table | **Did not exist** in any migration; code in 7 places referenced it | Created in migration `0006_dish_likes` with RLS + index | Without it the like feature couldn't survive a redeploy to a fresh Supabase project |
| 2 | `dish_comments` table | Did not exist; code wrapped in `try/catch` to silence the failure | Created in migration `0007_dish_comments` with RLS, **supports self-delete** | Same as above; plus original had no way to remove your own comment |
| 3 | RPC: `find_group_by_access_code` | Built manually in dashboard, not in migration | Created in migration `0010_rpc_functions` (`SECURITY DEFINER`, anon-revoked) | A redeploy would silently drop "Join Kitchen" |
| 4 | RPC: PIN verification | None — code did `password === input` plaintext compare client-side | New `verify_carte_password` + `set_carte_password` RPCs use pgcrypto `crypt()` server-side | PINs are now hashed; clients never send/receive plaintext after-the-fact |
| 5 | `profiles.role` | No role column; code inferred Chef/Diner by querying `menu_groups` | Explicit `role text` enum default `'both'` (`diner`/`chef`/`both`) | Removes a class of "is this user a chef?" race conditions and lets future role-aware UI consult one column |
| 6 | `menu_groups` privacy | `is_private` column was added but the code still used `password.trim().length > 0` as a fallback. Two sources of truth | `is_private` is the single source of truth; PIN moves to `password_hash` (nullable) | The dual-logic was actively leaking private cartes to the Discover feed |
| 7 | `menu_groups.password` storage | Plaintext `text` column | `password_hash text` set/verified through SECURITY DEFINER RPCs | Diners with read access could see PINs of other Diners' private cartes |
| 8 | `orders.tip` | `text` column | `numeric(10,2)` and **moved to `order_sessions`** | Was unaggregable; also "1 dish = 1 row, tip on every row" double-counted in the dashboard original |
| 9 | New `order_sessions` table | One order per dish row; no grouping; chef saw N independent toasts per checkout | Sessions group N orders, hold the single tip + notes per checkout | Matches the actual user mental model and lets the chef see a "ticket" |
| 10 | `orders.menu_group_id` (denormalized) | Required join `orders → dishes → categories → menu_groups` to filter by chef | Direct column with index on `(menu_group_id, status, created_at)` | Chef-orders query is now a single-table fetch; realtime filter `menu_group_id=eq.X` works without server-side join |
| 11 | `orders.price_at_order` (snapshot) | None — historical orders showed the dish's *current* price | Snapshot column populated at insert time | Chef can change a dish's price without rewriting history |
| 12 | `dishes.price` type | Text (originally) | `numeric(10,2)` with NOT NULL default 0 | Aggregable; arithmetic is correct |
| 13 | `dishes.recipe_private` semantics | Single column where the name was ambiguous (column name vs cell value) | Split into `recipe text` + `recipe_is_private boolean` | Clearer; lets Chef see their own private recipes; `useDishDetail` masks recipe for non-chef viewers |
| 14 | Wishlist vote dedup | `votes` was a plain integer; the same Diner could spam-vote | New `wishlist_votes(wishlist_id, user_id) PK` table; trigger maintains `wishlist.votes` count; toggle via `toggle_wishlist_vote` RPC | One vote per user per item, atomic toggle |
| 15 | Storage RLS scope | Policies named but `path` not constrained | Each user can only insert/update/delete `auth.uid()/*` paths in `menu-images` and `avatars` | Diners couldn't grief other users' images even if they had a public-bucket URL |
| 16 | Public buckets allow listing | Public `SELECT` policies allowed listing all objects | `SELECT` policies dropped; URL access still works (public bucket) | Surfaces fewer files than intended via storage API enumeration |
| 17 | Trigger function `search_path` mutability | `set_updated_at`, `menu_groups_normalize` didn't pin search_path | `ALTER FUNCTION ... SET search_path = public` | Closes a privilege-escalation vector; Supabase advisor was warning about it |

## Business logic fixes

| # | Item | Before | After | Why |
|---|---|---|---|---|
| 18 | Cart isolation | Single global cart `Context`; switching menus carried items across | `useCart` (Zustand) keys by `menu_group_id` (`cartByGroup: Record<groupId, items[]>`); `clearGroup(groupId)` on checkout | Diner can browse multiple kitchens without their cart contents bleeding |
| 19 | Realtime channel naming | `dish_likes_changes` (no user qualifier) | All channels now `<table>:<context>:<user.id>` | Avoids cross-user channel collision in multi-tenant scenarios |
| 20 | Discover feed pagination | `limit 100` on first load | `useInfiniteQuery`, 20 per page, near-bottom prefetch | Cheaper first paint; cursor pagination via `range()` |
| 21 | Discover feed realtime updates | Any `dish_likes` change forced a full feed refetch | `setQueryData` patches the affected dish's count + `liked_by_me` in place | One mutation no longer re-downloads 100 rows |
| 22 | Optimistic like updates | Single-cache (feed only) | `useToggleDishLike` updates BOTH feed cache AND `dish-detail` cache; rolls back both on error | Heart fills instantly on either screen |
| 23 | Order status notifications for Diner | None — Chef advanced status, Diner had no idea | `useRealtimeOrderStatus` mounted globally; toast on every `orders` UPDATE where `diner_id = me` | Closes the "is my order ready?" loop |
| 24 | Profile creation race | `auth.updateUser` could succeed while `profiles` upsert failed → orphaned auth user | `ensureSession` always followed by `ensureProfile` with insert-on-not-exists; trigger as a backup | App always boots into a usable state |
| 25 | Carte Code casing | Sometimes `toUpperCase()`, sometimes not, depending on path | Generated uppercase, normalized client-side, **trigger upper()** at DB level, **CHECK constraint** `^[A-Z0-9]{6}$` | Three-layer defense; can't get a lower-case row in |
| 26 | Join error feedback | Generic "Invalid password" for both wrong code and wrong PIN | Mutation throws typed errors (`NOT_FOUND` / `PIN_REQUIRED` / `WRONG_PIN`); UI surfaces a specific toast for each | User can tell which input was wrong |
| 27 | PIN verification | Client-side string compare | RPC `verify_carte_password` with bcrypt `crypt()` on the server; client only sees true/false | Hash never leaves DB; timing attacks aren't useful |

## Things deliberately NOT changed

These were original decisions that, on inspection, made sense — kept as-is:

- **Dual-role per user** ("both" mode): a user may be Chef in one carte and
  Diner in another. The original allowed this, and the new app's `role`
  column defaults to `'both'`.
- **PIN-protected (not full-auth) private cartes**: lightweight is the
  feature; not requiring everyone to sign up is what makes it usable.
- **Long-press to copy Carte Code**: cute, discoverable interaction kept
  intact.
- **2-column masonry on Discover**: visual signature.
- **3-second branded splash**: brand moment.
- **Brutalist palette** — single brown accent (`#A68B6A`) on grayscale,
  serif logo + system sans body. Mirrored exactly in `tailwind.config.js`
  and `lib/constants.ts`.
- **Bottom navigation height (64pt)** and 44pt minimum tap targets.
- **Carte Code 6-char alphanumeric** identifier shape.

## Things deferred (not in v1)

These were called out by the user as "if it needs manual config, skip":

- **Sign in with Apple / Google** — both require external OAuth setup
  (Apple Developer team, Google Cloud Console). The code is structured to
  add them via env-var feature flags later (`EXPO_PUBLIC_APPLE_AUTH_ENABLED`,
  `EXPO_PUBLIC_GOOGLE_CLIENT_ID`).
- **Push notifications** — needs APNs key uploaded to Expo. Order
  notifications work in foreground via realtime + haptics; backgrounded
  app currently misses them.
- **Privacy policy / Terms / Account-deletion screens** — Apple App Store
  will require them at submission time; planned as Phase J+1.
- **Chef-side cross-group wishlist FAB** — the per-group wishlist is in;
  the original's "look at all my cartes' wishlists at once" view is
  deferred (low value, easy to add).
- **Dark mode** — palette is light-only; original had dark mode but
  defaulted to light, so users never see it. Kept light for consistency.
